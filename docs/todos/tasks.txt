SKIP NOTE — Restore auditlog for Tenant + TenantSettings (2026-06-03)
----------------------------------------------------------------------

Cannot ship without forking django-auditlog. Root cause: the
``auditlog.LogEntry.actor`` field is a hard FK to
``settings.AUTH_USER_MODEL`` (= ``accounts.JasminUser``, which lives
in TENANT_APPS only). Moving the auditlog migrations into
SHARED_APPS so ``public.auditlog_logentry`` exists fails on the
foreign-key resolution: the FK target table
``public.accounts_jasminuser`` doesn't exist in the public schema —
JasminUser is tenant-scoped by design.

Confirmed via the failed attempt logged in audit_checklist.txt
Section 2 "Audit trail" under [PART] (2026-05-20 / rolled back same
day).

Three paths considered, none warranted:
  - Fork django-auditlog to make ``actor`` nullable + drop the FK.
    Heavy maintenance burden for a non-feature.
  - Build a parallel ``PublicAuditLogEntry`` model just for public-
    schema events. Doubles the audit machinery.
  - Accept the structured ``tenant.created`` / ``tenant.updated`` /
    ``user.roles_changed`` lines already going to auth.log as the
    forensics surface for Tenant + TenantSettings changes.

We chose path 3. The audit log gap is "no queryable diff for
Tenant/TenantSettings rows" — defensible because (a) those tables
change rarely (super-admin actions only), (b) the structured log
lines cover the "who did what when" question regulators ask, and
(c) the gap is documented openly in audit_checklist.txt.

Re-trigger if either:
  - django-auditlog ships a nullable-actor / generic-actor release
  - A tenant explicitly asks for queryable diff on these tables.

[ ] Off-site backups                                           TIER 2
    WHAT    Mirror every nightly encrypted pg_dump to an offsite
            location that lives outside the Linode prod VM's fate
            (separate provider, separate failure domain). Recovery
            target: even if the Linode VM + its disk + the Linode
            account are all gone, the backups survive.

    WHY     Today ``BACKUP_DIR=/backups`` is bind-mounted from the
            Linode VM's filesystem. Loses on:
              - Linode VM disk failure
              - Linode account compromise (attacker rm -rf's volume)
              - Ransomware that encrypts both the DB and /backups
            Auditor question this answers: "What's your DR plan?"

    ARCHITECTURE
            Two-copy story, not a move:
              Postgres on Linode VM
                   │ pg_dump → gzip → gpg encrypt
                   ▼
              /backups on Linode VM     ← BACKUP_DIR, GFS-pruned by
                                          the Huey ``prune_old_backups``
                                          task we already shipped
                   │ nightly rsync after each dump
                   ▼
              Hetzner Storage Box       ← independent provider,
                                          independent failure domain
            The local copy stays the fast-restore path. You only pull
            from Hetzner if the Linode copy is gone.

    RECOMMENDED  Hetzner Storage Box.
            Why: €3.20/mo for 1 TB, SFTP/SSH/rsync target out of the
            box, separate German provider (clean failure-domain
            separation from a US/Linode/AWS setup), no S3 dance.
            Alternative if you ever need true object-storage semantics:
            Hetzner Object Storage (S3-compatible, pay per GB).

    CONCRETE STEPS WHEN YOU'RE READY (~2 hours of actual work):

      1. Sign up for a Hetzner Storage Box (1 TB plan is fine).
         Note the host (e.g. ``u123456.your-storagebox.de``) and
         username.

      2. Generate a dedicated SSH keypair for backup uploads on a
         throwaway local machine:
           ssh-keygen -t ed25519 -N "" -f ./hetzner_backup_id_ed25519
         Upload the PUBLIC key to the Storage Box via Hetzner's web
         UI ("SSH keys"). Keep the private key in your password
         manager + drop it into a docker secret on prod.

      3. Add these to prod ``.env``:
           HETZNER_STORAGEBOX_HOST=u123456.your-storagebox.de
           HETZNER_STORAGEBOX_USER=u123456

      4. In ``docker-compose.yml`` for the ``backup`` service, mount
         the private key read-only:
           volumes:
             - ./backups:/backups:rw
             - ./secrets/hetzner_backup_id_ed25519:/etc/backup-ssh/id_rsa:ro

      5. Extend ``backups/backup.sh`` ``do_backup()`` with the rsync
         step at the very end:
           if [ -n "${HETZNER_STORAGEBOX_HOST:-}" ]; then
               rsync -av --delete-after \
                   -e "ssh -i /etc/backup-ssh/id_rsa \
                       -o StrictHostKeyChecking=accept-new \
                       -o UserKnownHostsFields=/dev/null" \
                   "$BACKUP_DIR/" \
                   "${HETZNER_STORAGEBOX_USER}@${HETZNER_STORAGEBOX_HOST}:backups/"
           fi
         ``--delete-after`` means the Hetzner copy tracks the local
         GFS-pruned set automatically — no separate offsite retention
         logic, the Huey task is still the single source of truth.

      6. Smoke test ONCE manually:
           docker compose exec backup /backup.sh now
         Confirm: (a) local /backups got a new file, (b) ``ssh ...
         storagebox.de ls backups/`` from a workstation sees it too.

      7. Restore drill from the Hetzner copy, per
         ``docs/security/restore-drill.md`` — pull the latest .sql.gz.gpg from
         Hetzner to a sandbox, replay it, assert row counts. This is
         the auditor-defensible "we proved the offsite chain works
         end-to-end" demo. Append the drill record to
         ``docs/security/restore-drills/``.

      8. Flip the audit_checklist.txt Section 3 "Backups" entries
         (currently ``[GAP] Backups are local-only``) to ``[OK]`` with
         a pointer to this section + the first drill report.

    WHAT STAYS THE SAME
      - ``BACKUP_DIR`` stays ``/backups`` (local copy is the fast
        path).
      - GPG encryption stays mandatory (Hetzner staff can read disk
        contents; AES-256 with your key means they can't).
      - The Huey ``prune_old_backups`` task stays local-only. The
        rsync ``--delete-after`` propagates its decisions offsite for
        free.
      - No Django code change. This entire item is shell + docker-
        compose + env-var work.

    GOTCHAS WORTH KNOWING
      - First rsync after a new GPG key rotation will re-upload
        EVERY file (ciphertext changes). Expect a one-night bandwidth
        spike on key rotations.
      - StrictHostKeyChecking=accept-new only accepts on first
        connection. Pin the actual host key into the ssh config file
        for true MITM resistance once you've verified the fingerprint
        against Hetzner's official one.
      - Don't lose the GPG passphrase + the SSH private key. They're
        the only things between the ciphertext on Hetzner and a
        readable backup. Treat them with the same discipline as the
        DJANGO_SECRET_KEY rotation runbook.

    EFFORT  2 hours of actual coding once you have Hetzner credentials.
            Sign-up + key generation + first-rsync verification adds
            another ~30 min.

HUEY CONVENTIONS (preserved from old huey-to-do.txt)
==================================================================

Production runs under docker-compose with a dedicated ``huey``
service (see docker-compose.yml). Scheduled work belongs there,
NOT in host cron / a sidecar cron container / cron-inside-the-
backend-container. Reasons:

  - One scheduler to run, monitor, and reason about.
  - Tasks share the Django app context (ORM, settings, logging,
    auditlog).
  - Failures land in the Huey worker logs alongside the worker's
    other output — one log to grep instead of two.
  - No host coupling: redeploys don't drift away from a crontab
    the person who set it up forgot to commit.

Conventions
-----------
  - File:   ``apps/<app>/tasks.py`` (Huey autodiscovers this name
            via ``huey.contrib.djhuey``).
  - Decor:  ``@db_periodic_task(crontab(...))`` from
            ``huey.contrib.djhuey``.
  - Multi-tenant: tasks that touch tenant-scoped tables MUST
            iterate via ``Tenant.objects.exclude(schema_name="public")``
            and use ``schema_context(tenant.schema_name)`` per tenant.
  - Logging: log via ``logging.getLogger(...)``; the huey service's
            stdout goes to its container log.
  - Return value: if you want
            ``python manage.py run_periodic_tasks_now`` to print a
            one-line observation summary, return a flat
            ``dict[str, int]`` from the task body — the command
            formats it as ``key=value key=value``. Returning ``None``
            is fine; the runner just prints no summary.

Bootstrap
---------
Before any of these can run, uncomment + adapt the ``HUEY`` config
block in ``config/settings.py``. Switch ``huey_class`` from
``SqliteHuey`` to ``RedisHuey`` — Redis already runs in
docker-compose. Confirm ``"periodic": True`` in the consumer block.

To register a new periodic task in the dev/QA runner, add it to
the ``TASK_REGISTRY`` in
apps/shared/tenants/management/commands/run_periodic_tasks_now.py.


==================================================================
WHAT STAYS OUT OF HUEY
==================================================================

  - pg_dump backups themselves. Run dumps from a sidecar container
    with read-only DB credentials, scoped to write-only access to
    the backups volume; keep that container small and single-purpose.
    (Backup pruning IS in Huey — that's a different concern.)
  - TLS certificate renewal (already handled by certbot in
    docker-compose).
  - Anything that needs to run when the Huey worker is down
    (e.g. health checks for the worker itself).
