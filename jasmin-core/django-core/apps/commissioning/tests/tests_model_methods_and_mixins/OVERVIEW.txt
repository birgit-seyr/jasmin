test_order_content_resolvers.py
-------------------------------
Source: apps/commissioning/models/resellers.py (OrderContent.resolve_*)
DB required: Yes (uses factories)

Lock-down for the SINGLE source of truth for resolving an
OrderContent's effective share_article and amount_per_pu. unit and
size are direct OrderContent columns (NOT resolved) — when an OC is
created from an offer the service copies offer.unit / offer.size onto
the row, so the on-row value is always populated.

Model invariant: OrderContent has EXACTLY ONE of (offer,
share_article) populated. The resolver's precedence between the two
is therefore academic — only one branch is ever live on a persisted
row.

If any of these tests fail, the regression is almost certainly that
someone reintroduced an inline ``offer.X if offer else content.X``
pattern somewhere — fix the caller (route through resolve_*), don't
loosen the test.

  TestExactlyOneItemTypeInvariant
    - clean() raises when both offer and share_article set
    - clean() raises when neither offer nor share_article set

  TestResolveShareArticle
    - offer-bound OC → returns offer.share_article
    - ad-hoc OC (no offer) → returns content.share_article

  TestResolveAmountPerPu
    - offer.amount_per_pu wins
    - no-offer ad-hoc: falls back to article.get_amount_per_pu_for_reseller(content.unit) for KG/PCS
    - falls back to 1 when article has no value for the line's unit
    - offer.amount_per_pu == 0 is falsy and falls through to offer's article

  TestLineNettoIsUnitBased
    - line_netto = amount × price_per_unit (amount is in UNITS, price is
      €/unit); amount_per_pu does NOT enter pricing (DOC-1/DOC-2 fix)
    - ad-hoc OC and offer-bound OC both ignore amount_per_pu for the net,
      keeping order == delivery-note == invoice line totals

  TestCommissioningListUsesResolver
    - _build_content_entry returns article's amount_per_pu when no offer
    - _build_content_entry returns offer's amount_per_pu when offer set
